Eightfold Assignment — Phase-Wise Development Plan
This is a two-deliverable assignment (Step 1: design doc, Step 2: working code), evaluated on correctness, reasoning, edge-case handling, and explainability — not feature count. Below is a phase breakdown you can follow end-to-end, sized for an intern-level submission that still reads as production-minded.
Phase 0 — Scoping & Tech Decisions (before any design doc text)

Pick language: Python is the strongest fit here — phonenumbers for E.164, python-dateutil/dateparser for messy dates, pdfplumber/python-docx for resumes, requests for GitHub's REST API, jsonschema/pydantic for validation, click/argparse for the CLI.
Pick your two-plus sources now, since it determines scope: recommend Recruiter CSV + ATS JSON (structured) and GitHub API + Resume PDF (unstructured) — GitHub has a real public API (no auth headaches), and resumes exercise real parsing logic. LinkedIn scraping is fragile/ToS-risky, so avoid it unless you mock the data.
Decide upfront what's explicitly out of scope (e.g., live LinkedIn scraping, OCR for scanned resumes, NER-based skill extraction beyond a keyword/alias dictionary) — write this down now so it lands cleanly in the README's "descoped" section later.

Phase 1 — Technical Design (Step 1 deliverable)
This is a one-page PDF, no code, named <YourFullName>_<YourEmail>_Eightfold.pdf. Cover exactly what's asked, tightly:

Pipeline diagram: your own step breakdown (e.g., ingest → detect_source_type → extract → normalize → entity_resolve/merge → score_confidence → project(config) → validate → emit). Don't just copy the dummy pipeline — note one or two things you changed and why (e.g., splitting "merge" and "confidence" into separate passes so confidence can see cross-source agreement).
Canonical schema + normalization choices: confirm the default schema fields, and state your concrete normalization rules — phone → E.164 via phonenumbers with a default region fallback, dates → YYYY-MM (handle "Present"/ongoing roles explicitly), country → ISO-3166 alpha-2 via a lookup table, skills → canonicalized via a synonym map (e.g., "ReactJS"/"React.js"/"react" → "React").
Merge/conflict policy: define your match key (email is primary identity key; fallback to normalized full_name + phone if email absent), and your winner-selection rule (e.g., source-priority ranking like ATS > Resume > GitHub > Notes for title/company, but "most recent timestamp wins" for things like headline; majority-vote for skills appearing in 2+ sources).
Confidence model: a simple, explainable formula beats a fancy opaque one — e.g., confidence = base_by_source_type + agreement_bonus(if 2+ sources concur) - staleness_penalty, capped at 1.0. State it as a formula, not just "high/medium/low."
Config handling: explain the separation between your internal canonical record (full fidelity) and a stateless projection function that reads the config and emits a subset/renamed/normalized view — and that you validate the projection output against a JSON Schema generated from the config's field list.
3–5 edge cases + what's deliberately left out: pick concrete ones — empty/malformed CSV row, GitHub user with 0 public repos, two sources disagreeing on current company, resume with unparseable dates, candidate appearing in 3 sources with 3 different name spellings. Then one explicit line on what you cut (e.g., "no OCR for scanned PDFs — falls back to null + low confidence").

Phase 2 — Core Data Model & Schema

Define the canonical internal model in code (dataclasses or Pydantic models) matching the table exactly: candidate_id, full_name, emails[], phones[], location{}, links{}, headline, years_experience, skills[], experience[], education[], provenance[], overall_confidence.
Write the canonical JSON Schema for the default output once, and make sure your projection/validation layer can derive a schema for any custom config from the same field definitions — don't hand-write two separate schemas.
Decide candidate_id generation strategy (deterministic hash of normalized email, or UUID assigned at first-sight) — document it, since "deterministic same-input-same-output" is an explicit constraint.

Phase 3 — Source Extractors (one per source type)

Build a thin adapter interface, e.g. extract(raw_input) -> RawRecord, so every source type implements the same contract regardless of how messy it is internally.
CSV extractor: tolerant row parsing (skip/flag malformed rows, don't crash), header normalization.
ATS JSON extractor: explicit field-name remapping table from ATS's own keys to canonical keys (this is literally listed as "field names that do NOT match ours" — make the mapping visible/configurable, not hardcoded inline).
GitHub extractor: call the public REST API for profile + repo languages; degrade gracefully on rate-limit/404 (empty profile ≠ crash).
Resume extractor: pull raw text via pdfplumber/python-docx, then a light rule-based parser for sections (Experience/Education/Skills headers) — be honest in the design doc that prose parsing is heuristic, not NLP-grade.
Every extractor returns the same intermediate shape plus a source_type tag, so downstream code never special-cases per-source logic.

Phase 4 — Normalization Layer

One normalizer function per field type (phone, date, country, skill), each pure and independently testable.
Skills: maintain a small canonical alias dictionary (JSON file you can extend), normalize casing/whitespace before lookup, and let unknowns pass through as their cleaned form rather than being dropped — but flag them with lower confidence.
Dates: handle missing day/month gracefully, "Present"/"Current" as open-ended, and unparseable strings → null (never invented).

Phase 5 — Identity Resolution & Merge Engine

Implement the match-key strategy from your design doc (email primary, name+phone fallback) as a single, testable function — this is the highest-leverage piece of the whole assignment to get right and explain well.
Conflict resolution per field: implement your source-priority table as data (a dict), not as scattered if/else, so it's auditable and easy to defend in the demo video.
Populate provenance[] at merge time — every written field records {field, source, method} so nothing is untraceable.

Phase 6 — Confidence Scoring

Implement the formula from the design doc as one small, well-named function per field, then aggregate to overall_confidence (e.g., weighted average, or min-of-critical-fields — pick one and justify it).
Make sure confidence visibly drops for single-source, stale, or heuristically-parsed (resume prose) fields — this is what shows the "wrong-but-confident is worse than empty" principle from the brief in action.

Phase 7 — Projection Layer (config-driven output)

Parse the runtime config (field list with path/from/type/normalize/required, plus include_confidence/include_provenance/on_missing).
Implement projection as a pure function project(canonical_record, config) -> dict, fully decoupled from extraction/merge — this is the "clean separation" the brief explicitly calls out as a constraint.
Implement on_missing modes precisely: null (write null), omit (drop the key), error (raise/report a clear validation error) — test all three.

Phase 8 — Validation

Validate every projected output against a schema derived from the active config before returning it (use jsonschema against a dynamically built schema, or Pydantic models generated per-config).
Validation failures should produce actionable error messages, not stack traces — and should never crash the whole batch run (one bad candidate shouldn't kill thousands of others — this maps directly to the "Scale" constraint).

Phase 9 — CLI / Thin I/O Surface

A single CLI entrypoint, e.g. python run.py --csv recruiter.csv --json ats.json --github usernames.txt --resumes ./resumes/ --config config.json --out output.json.
Keep it intentionally minimal per the brief ("don't spend time on polish here") — argument parsing, clear stdout logging of skipped/malformed rows, and a JSON file written to disk is enough. Skip a UI unless you have spare time at the end.

Phase 10 — Testing & Edge Cases (optional but recommended)

Unit tests per normalizer (phone, date, skill) and per merge-conflict scenario.
At least one gold-profile test: hand-craft 2–3 source files for one fictional candidate with deliberate conflicts, assert the merged+projected output matches an expected JSON fixture exactly — this single test does the most to demonstrate determinism and correctness in your evaluation.
One explicit edge-case test: empty/garbage source file → pipeline still completes and returns nulls, not a crash.

Phase 11 — Documentation

README: exact run steps, sample command, where output lands, list of which 2+ source types you implemented, and an explicit "Assumptions & Descoped" section (mirrors what you wrote in the design doc — keep them consistent, reviewers will cross-check).
Include the actual produced output JSON (default config + one custom config) in the repo, not just instructions to regenerate it.

Phase 12 — Demo Video & Submission Packaging

~2 minutes: run end-to-end on sample inputs, show default-schema output, show one custom-config output, then talk through one design decision you're proud of (e.g., your confidence formula or merge-priority table) and one edge case you handled — these two talking points are explicitly what's evaluated, so script them rather than improvising live.
Final checklist before submitting: design PDF named exactly as specified, public GitHub repo with README + output + tests, video link, and a final sanity run from a clean clone to make sure "exact run steps" actually work for someone else.
